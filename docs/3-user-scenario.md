## 1. 시나리오 개요

본 문서는 TodoListApp의 주요 사용자 시나리오를 Use Case 단위로 기술합니다. 각 시나리오는 정상 흐름, 대안 흐름, 예외 흐름으로 구성되어 개발팀과 QA팀이 테스트 케이스를 도출할 수 있도록 작성되었습니다.

### 대상 페르소나

| 페르소나 | 연령 | 직업 | 주요 사용 장치 | 주요 요구사항 |
|---------|------|------|----------------|--------------|
| 김지우 | 34세 | 마케터 | 회사 PC(Chrome), 스마트폰 | 카테고리별 업무/개인 구분 |
| 박서준 | 27세 | 대학원생 | 맥북(Safari/Chrome), 스마트폰 | 프로젝트별 분류 및 진척률 확인 |

### MVP 사용 사례 매트릭스

| UC ID | 기능명 | 우선순위 | 1차 포함 | 관련 화면 | 관련 API |
|-------|--------|---------|--------|---------|----------|
| UC-01 | 회원가입 | High | O | SCR-01 | POST /api/auth/register |
| UC-02 | 로그인 | High | O | SCR-02 | POST /api/auth/login |
| UC-05 | 할일 등록 | High | O | SCR-04 | POST /api/todos |

## UC-01: 회원가입

### 2.1 기본 정보

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-01 |
| **기능명** | 회원가입 |
| **액터** | 비인증 사용자 |
| **우선순위** | High |
| **화면** | SCR-01 (회원가입 화면) |
| **API** | POST /api/auth/register |

### 2.2 사전 조건

- 사용자가 애플리케이션 접속 후 비인증 상태임
- 시스템이 정상 운영 중

### 2.3 정상 흐름 (Main Flow)

#### MC-01-01: 신규 사용자 회원가입 완료

| 단계 | 액터(사용자) 행동 | 시스템 응답 | 관련 규칙 |
|------|-------------------|-----------|----------|
| 1 | 회원가입 화면 이동 (SCR-01) | SCR-01 렌더링 (이메일, 비밀번호, 이름 입력 필드 및 가입 버튼 표시) | - |
| 2 | 이메일 입력 필드에 "user@example.com" 입력 | 입력 필드에 텍스트 반영 | BR-01: 이메일 형식 검증 |
| 3 | 비밀번호 입력 필드에 "MyPassword123!" 입력 | 입력 필드에 마스킹 처리되어 표시 | BR-02: 비밀번호 bcrypt 해시 저장 |
| 4 | 이름 입력 필드에 "김지우" 입력 | 입력 필드에 텍스트 반영 | - |
| 5 | "가입 완료" 버튼 클릭 | POST /api/auth/register 호출 (body: {email, password, name}) | - |
| 6 | 서버는 요청받은 이메일이 미존재 확인 | 서버 내부 유효성 검사 실행 | BR-01: 이메일 중복 확인 |
| 7 | 서버는 비밀번호를 bcrypt로 해시 처리 | 서버 내부 암호화 | BR-02: bcrypt 해시 처리 |
| 8 | 서버는 users 테이블에 새 레코드 삽입 | 계정 생성 완료, HTTP 201 Created 응답 | - |
| 9 | 클라이언트는 성공 메시지 표시 후 로그인 화면으로 리다이렉트 | SCR-02 (로그인 화면) 자동 이동 | - |

**사후 조건:**
- 신규 사용자 계정 생성 완료
- users 테이블에 새로운 행 추가됨
- 클라이언트가 로그인 화면으로 이동

**관련 데이터:**
```sql
-- users 테이블 신규 행
INSERT INTO users (email, password, name, created_at)
VALUES ('user@example.com', bcrypt_hash, '김지우', NOW());
```

## UC-02: 로그인

### 2.6 기본 정보

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-02 |
| **기능명** | 로그인 |
| **액터** | 비인증 사용자 (기존 계정 보유) |
| **우선순위** | High |
| **화면** | SCR-02 (로그인 화면) |
| **API** | POST /api/auth/login |

### 2.7 사전 조건

- 사용자가 이미 회원가입된 계정 보유 (UC-01 완료)
- 사용자가 비인증 상태
- 시스템이 정상 운영 중

### 2.8 정상 흐름 (Main Flow)

#### MC-02-01: 기존 사용자 로그인 성공

| 단계 | 액터(사용자) 행동 | 시스템 응답 | 관련 규칙 |
|------|-------------------|-----------|----------|
| 1 | 로그인 화면 접속 (SCR-02) | SCR-02 렌더링 (이메일, 비밀번호 입력 필드 및 로그인 버튼 표시) | - |
| 2 | 이메일 입력 필드에 "kim@example.com" 입력 | 입력 필드에 텍스트 반영 | - |
| 3 | 비밀번호 입력 필드에 "MyPassword123!" 입력 | 입력 필드에 마스킹 처리되어 표시 | - |
| 4 | "로그인" 버튼 클릭 | POST /api/auth/login 호출 (body: {email, password}) | - |
| 5 | 서버는 users 테이블에서 해당 이메일의 사용자 검색 | 사용자 레코드 조회 | BR-03: 이메일/비밀번호 검증 |
| 6 | 서버는 입력받은 비밀번호와 저장된 bcrypt 해시 비교 | 일치 확인 | BR-02: bcrypt 해시 검증 |
| 7 | 서버는 JWT Access Token 생성 (payload: user_id, email) | 토큰 생성 완료 | BR-03: JWT 발급 |
| 8 | 서버는 HTTP 200 OK 응답과 토큰 반환 | 토큰을 클라이언트에 전달 | - |
| 9 | 클라이언트는 토큰을 Zustand 스토어(메모리)에 저장 | 토큰 저장 완료 | - |
| 10 | 클라이언트는 할일 목록 화면으로 리다이렉트 | SCR-03 (할일 목록 화면) 렌더링 | - |

**사후 조건:**
- 사용자가 인증된 상태로 전환
- 클라이언트에 JWT Access Token 저장
- 클라이언트가 할일 목록 화면으로 이동
- 이후 모든 API 요청 시 Authorization 헤더에 토큰 포함

**관련 데이터:**
```json
{
  "status": 200,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "token_type": "Bearer",
    "user": {
      "id": 1,
      "email": "kim@example.com",
      "name": "김지우"
    }
  }
}
```

## UC-05: 할일 등록

### 2.11 기본 정보

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-05 |
| **기능명** | 할일 등록 |
| **액터** | 인증된 사용자 |
| **우선순위** | High |
| **화면** | SCR-04 (할일 등록 화면) |
| **API** | POST /api/todos |

### 2.12 사전 조건

- 사용자가 로그인 완료 (UC-02 완료, JWT Access Token 보유)
- 할일 목록 화면(SCR-03)에서 "새 할일 추가" 버튼 클릭 또는 직접 할일 등록 화면 접속
- 시스템이 정상 운영 중
- 기본 카테고리(업무, 개인, 쇼핑 등)가 DB에 존재

### 2.13 정상 흐름 (Main Flow)

#### MC-05-01: 필수 정보만 입력하여 할일 등록

| 단계 | 액터(사용자) 행동 | 시스템 응답 | 관련 규칙 |
|------|-------------------|-----------|----------|
| 1 | 할일 등록 화면 접속 (SCR-04) | SCR-04 렌더링 (제목, 설명, 종료예정일, 카테고리 선택 입력 필드 및 저장 버튼 표시) | - |
| 2 | 제목 필드에 "마케팅 전략서 작성" 입력 | 입력 필드에 텍스트 반영 | BR-08: 제목 필수입력 |
| 3 | 설명 필드 공란 유지 (선택사항) | 입력 필드 공란 유지 | - |
| 4 | 종료예정일 필드 공란 유지 (선택사항) | 입력 필드 공란 유지 | - |
| 5 | 카테고리 드롭다운에서 "업무" 선택 | 카테고리 "업무" 선택됨 | BR-07: 카테고리 선택 |
| 6 | "저장" 버튼 클릭 | POST /api/todos 호출 (body: {title, category_id, description: null, due_date: null}) | - |
| 7 | 서버는 Authorization 헤더의 JWT 검증 | 사용자 인증 완료 | BR-03, BR-09: 인증 & 소유권 확인 |
| 8 | 서버는 todos 테이블에 새 할일 행 삽입 | 할일 생성 완료, HTTP 201 Created 응답 | - |
| 9 | 클라이언트는 성공 메시지 표시 후 할일 목록 화면으로 리다이렉트 | SCR-03 (할일 목록 화면) 렌더링, 새 할일 목록에 표시 | - |

**사후 조건:**
- 새 할일이 todos 테이블에 저장됨
- 사용자의 할일 목록 화면에 새 할일 표시
- is_completed = FALSE (기본값)

**관련 데이터:**
```sql
-- todos 테이블 신규 행
INSERT INTO todos (user_id, category_id, title, description, due_date, is_completed, created_at, updated_at)
VALUES (1, 1, '마케팅 전략서 작성', NULL, NULL, FALSE, NOW(), NOW());
```

#### MC-05-02: 모든 정보를 입력하여 할일 등록

| 단계 | 액터(사용자) 행동 | 시스템 응답 | 관련 규칙 |

## 4. 크로스 시나리오 테스트 케이스

### 4.1 데이터 접근 제어 (Authorization) 테스트

**TC-AUTH-01: 다른 사용자의 할일 조회 불가**
```
1. 사용자 A(user_id=1)로 로그인 → JWT Token A 획득
2. 사용자 B(user_id=2)의 할일을 조회 요청: GET /api/todos/100 (user_id=2의 할일)
3. Authorization 헤더에 Token A 포함
4. 기대 결과: HTTP 403 Forbidden 또는 404 Not Found 응답
```

**TC-AUTH-02: 자신의 할일만 조회 가능**
```
1. 사용자 A(user_id=1)로 로그인 → JWT Token A 획득
2. 사용자 A의 할일 조회: GET /api/todos
3. 기대 결과: HTTP 200, user_id=1의 할일만 반환
```

---

### 4.2 카테고리 관리 테스트

**TC-CAT-01: 기본 카테고리는 모든 사용자가 접근 가능**
```
1. 사용자 A로 로그인, GET /api/categories 호출
2. 사용자 B로 로그인, 동일하게 GET /api/categories 호출
3. 기대 결과: 두 사용자 모두 동일한 기본 카테고리 목록 반환
   - 예: [업무, 개인, 쇼핑, 기타]
```

---

### 4.3 기간 초과 판별 테스트

**TC-DUE-01: 오늘보다 과거인 미완료 할일이 기간 초과로 표시**
```
1. 할일 생성: title="과제", due_date="2026-05-10", is_completed=false (today=2026-05-13)
2. GET /api/todos?overdue=true 호출
3. 기대 결과: HTTP 200, 해당 할일 반환
4. UI: 해당 할일이 빨간색으로 강조
```

**TC-DUE-02: 오늘 이후 미완료 할일은 기간 초과 아님**
```
1. 할일 생성: title="과제", due_date="2026-05-20", is_completed=false (today=2026-05-13)
2. GET /api/todos?overdue=true 호출
3. 기대 결과: HTTP 200, 빈 배열 반환
```

**TC-DUE-03: 완료된 할일(과거 due_date)은 기간 초과 대상 아님**
```
1. 할일 생성: title="과제", due_date="2026-05-10", is_completed=true
2. GET /api/todos?overdue=true 호출
3. 기대 결과: HTTP 200, 해당 할일 미반환 (완료되었으므로)
```

---

### 4.4 필터링 조합 테스트

**TC-FILTER-01: 카테고리 + 완료 여부 복합 필터**
```
1. GET /api/todos?category_id=1&is_completed=false
2. 기대 결과: category_id=1 AND is_completed=FALSE 조건 만족 할일만 반환
```

**TC-FILTER-02: 카테고리 + 기간 초과 복합 필터**
```
1. GET /api/todos?category_id=1&overdue=true
2. 기대 결과: category_id=1 AND due_date < TODAY AND is_completed=FALSE 조건 만족 할일만 반환
```

**TC-FILTER-03: 카테고리 + 완료 여부 + 기간 초과 삼중 필터**
```
1. GET /api/todos?category_id=1&is_completed=true&overdue=true
2. 기대 결과: 조건 만족 할일 반환 (과거 due_date이면서 완료된 할일)
```

## 6. 성공 기준 (Acceptance Criteria)

### 6.1 각 UC별 성공 기준

| UC | 성공 기준 |
|----|---------|
| UC-01 (회원가입) | 신규 이메일로 가입 완료, 로그인 화면으로 자동 리다이렉트, 로그인 가능 |
