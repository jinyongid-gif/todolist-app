# TodoListApp 도메인 정의서

**버전:** 1.0
**작성일:** 2026-05-12
**작성자:** Business Analyst

---

## 1. 프로젝트 개요

### 프로젝트명
TodoListApp

### 목적
인증된 사용자가 자신만의 할일을 카테고리별로 체계적으로 관리할 수 있는 개인 할일 관리 웹 애플리케이션을 제공한다.

### 문제 정의

| 문제 | 설명 |
|------|------|
| 개인화된 할일 관리 부재 | 사용자별로 분리된 개인 할일 관리 환경을 제공하는 앱이 없다 |
| 인증 기반 데이터 저장 미흡 | 데이터베이스에 인증된 사용자 단위로 할일을 저장하고 관리하는 앱이 없다 |
| 카테고리 분류 기능 부재 | 할일을 카테고리별로 등록하고 필터링할 수 있는 앱이 없다 |

---

## 2. 핵심 도메인 개념 (유비쿼터스 언어)

| 용어 | 설명 |
|------|------|
| **사용자 (User)** | 회원가입 후 인증을 통해 앱을 사용하는 주체 |
| **인증 (Authentication)** | 사용자가 이메일과 비밀번호로 신원을 확인하는 과정 |
| **할일 (Todo)** | 사용자가 수행해야 할 작업 단위 |
| **카테고리 (Category)** | 할일을 분류하기 위한 그룹 단위. 기본 카테고리와 사용자 정의 카테고리로 구분됨 |
| **기본 카테고리 (Default Category)** | 시스템에서 모든 사용자에게 기본 제공되는 카테고리 |
| **사용자 정의 카테고리 (Custom Category)** | 인증된 사용자가 직접 생성한 카테고리 |
| **종료예정일 (Due Date)** | 할일을 완료하기로 계획한 목표 날짜 |
| **완료 여부 (Completion Status)** | 할일이 완료되었는지 여부를 나타내는 상태값 |
| **기간 초과 (Overdue)** | 종료예정일이 현재 날짜를 지났으나 완료되지 않은 상태 |

## 3. 도메인 모델

### 핵심 엔티티와 속성

#### User (사용자)
- `id` : 고유 식별자 (PK)
- `email` : 이메일 주소 (로그인 ID, 유일값)
- `password` : 암호화된 비밀번호
- `name` : 이름
- `created_at` : 가입일시

#### Category (카테고리)
- `id` : 고유 식별자 (PK)
- `name` : 카테고리명
- `is_default` : 기본 카테고리 여부
- `user_id` : 소유 사용자 ID (기본 카테고리는 NULL)

#### Todo (할일)
- `id` : 고유 식별자 (PK)
- `user_id` : 소유 사용자 ID (FK)
- `category_id` : 카테고리 ID (FK)
- `title` : 할일 제목
- `description` : 설명
- `due_date` : 종료예정일
- `is_completed` : 완료 여부
- `created_at` : 등록일시
- `updated_at` : 최종 수정일시

### 엔티티 간 관계

- **User -- Category** : 사용자는 0개 이상의 사용자 정의 카테고리를 소유한다 (1:N)
- **User -- Todo** : 사용자는 0개 이상의 할일을 소유한다 (1:N)
- **Category -- Todo** : 카테고리는 0개 이상의 할일을 포함한다 (1:N)
- **기본 카테고리** : 특정 사용자에게 종속되지 않으며 모든 사용자가 공통으로 사용한다

## 4. 핵심 비즈니스 규칙

### 사용자 규칙
- BR-01 : 이메일은 시스템 전체에서 유일해야 한다
- BR-02 : 비밀번호는 암호화하여 저장해야 한다
- BR-03 : 인증되지 않은 사용자는 할일 및 카테고리에 접근할 수 없다

### 카테고리 규칙
- BR-04 : 기본 카테고리는 시스템이 제공하며 사용자가 삭제할 수 없다
- BR-05 : 사용자 정의 카테고리는 해당 사용자만 접근하고 수정할 수 있다
- BR-06 : 카테고리명은 동일 user_id 범위 내에서 중복될 수 없다. 기본 카테고리(user_id=NULL)끼리도 name 중복 불가

### 할일 규칙
- BR-07 : 할일은 반드시 하나의 카테고리에 속해야 한다
- BR-08 : 할일 제목은 필수 입력값이다
- BR-09 : 사용자는 자신의 할일만 조회, 수정, 삭제할 수 있다
- BR-10 : 완료 처리된 할일은 완료 취소(되돌리기)가 가능하다

## 5. 주요 유스케이스

### 액터
- **비인증 사용자** : 회원가입 및 로그인만 수행 가능
- **인증된 사용자** : 개인정보 수정, 할일 및 카테고리 전체 기능 수행 가능

### 유스케이스 목록

| ID | 유스케이스 | 액터 | 주요 흐름 요약 |
|----|-----------|------|----------------|
| UC-01 | 회원가입 | 비인증 사용자 | 이메일, 비밀번호, 이름 입력 후 계정 생성 |
| UC-02 | 로그인 | 비인증 사용자 | 이메일, 비밀번호 입력 후 인증 토큰 발급 |
| UC-03 | 개인정보 수정 | 인증된 사용자 | 이름, 비밀번호 등 개인정보 변경 |
| UC-04 | 카테고리 추가 | 인증된 사용자 | 카테고리명 입력 후 사용자 정의 카테고리 생성 |
| UC-05 | 할일 등록 | 인증된 사용자 | 제목, 설명, 종료예정일, 카테고리 선택 후 저장 |
| UC-06 | 할일 목록 조회 | 인증된 사용자 | 카테고리, 기간 초과 여부, 완료 여부 기준으로 필터링하여 목록 표시 |
| UC-07 | 할일 수정 | 인증된 사용자 | 등록된 할일의 정보 변경 |
| UC-08 | 할일 완료 처리 | 인증된 사용자 | 할일을 완료 상태로 변경 (취소 가능) |
| UC-09 | 할일 삭제 | 인증된 사용자 | 선택한 할일을 영구 삭제 |

## 6. 도메인 경계 (바운디드 컨텍스트)

### 인증 컨텍스트 (Authentication Context)
- **범위** : 회원가입, 로그인, 인증 토큰 관리
- **핵심 관심사** : 사용자 신원 확인 및 접근 제어
- **주요 엔티티** : User (인증 정보 관점)

### 사용자 프로필 컨텍스트 (User Profile Context)
- **범위** : 개인정보 조회 및 수정
- **핵심 관심사** : 사용자 프로필 데이터 관리
- **주요 엔티티** : User (프로필 정보 관점)

### 할일 관리 컨텍스트 (Todo Management Context)
- **범위** : 할일 등록, 조회, 수정, 삭제, 완료 처리, 필터링
- **핵심 관심사** : 개인화된 할일 데이터의 CRUD 및 상태 관리
- **주요 엔티티** : Todo, Category

### 컨텍스트 간 의존성
- 할일 관리 컨텍스트는 인증 컨텍스트에 의존한다 (인증된 사용자만 접근 가능)
- 사용자 프로필 컨텍스트는 인증 컨텍스트에 의존한다 (인증된 사용자만 수정 가능)
