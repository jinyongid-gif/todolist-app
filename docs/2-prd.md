## 1. 제품 개요

### 한 줄 제품 목적
인증된 사용자가 카테고리별로 할일을 체계적으로 등록·관리할 수 있는 개인 할일 관리 웹 애플리케이션

### 해결하는 문제

| 문제 | 설명 |
|------|------|
| 개인화된 할일 관리 부재 | 사용자별로 완전히 분리된 개인 할일 관리 환경이 없다 |
| 인증 기반 데이터 저장 미흡 | 데이터베이스에 인증된 사용자 단위로 할일을 안전하게 저장·관리하는 수단이 없다 |
| 카테고리 분류 기능 부재 | 할일을 카테고리별로 등록하고 필터링할 수 있는 통합 도구가 없다 |

### 핵심 가치 제안
- 로그인 한 번으로 내 할일만 보이는 **안전한 개인 공간** 제공
- 기본 카테고리와 사용자 정의 카테고리를 통한 **유연한 할일 분류**
- 종료예정일·완료 여부·기간 초과 기준 필터링으로 **빠른 현황 파악**

## 2. 타겟 사용자

### 페르소나 1 — 바쁜 직장인 김지우 (34세, 마케터)

- **불편함**: 업무·개인 할일이 뒤섞여 무엇부터 해야 할지 파악하기 어렵다. 별도 앱 없이 메모장이나 채팅방 즐겨찾기로 관리하다 놓치는 일이 잦다.
- **목표**: 카테고리(업무, 개인, 쇼핑 등)로 할일을 구분하고 종료예정일에 맞춰 우선순위를 확인하고 싶다.
- **사용 환경**: 회사 PC(Chrome), 퇴근 후 스마트폰(Mobile Chrome/Safari)

### 페르소나 2 — 자기계발 중인 대학원생 박서준 (27세, 대학원생)

- **불편함**: 논문·수업·아르바이트 일정이 산재해 있고 완료한 항목과 미완료 항목을 한눈에 비교하기 힘들다.
- **목표**: 프로젝트별로 할일을 분류하고 완료 여부를 빠르게 체크·취소하면서 진척률을 시각적으로 확인하고 싶다.
- **사용 환경**: 맥북(Safari/Chrome), 강의실 이동 중 스마트폰(Mobile Chrome)

### 사용 환경 요약

| 구분 | 내용 |
|------|------|
| 플랫폼 | Web + Mobile Web (반응형 웹 UI) |
| 주요 브라우저 | Chrome, Safari (최신 2개 메이저 버전) |
| 우선 최적화 뷰포트 | 모바일(375px~) → 태블릿(768px~) → 데스크톱(1280px~) |

## 3. 기능 요구사항

### MVP 분류 기준

> **MVP**: 앱을 최소한으로 동작시키는 핵심 기능 — 인증, 할일 CRUD, 완료 처리, 카테고리 기반 필터링
>
> **Post-MVP**: 없어도 앱을 사용할 수 있는 편의 기능 — 개인정보 수정, 사용자 정의 카테고리 관리

### 3.1 MVP 기능 (1차 릴리즈 — 3일)

| UC | 기능명 | 우선순위 | 핵심 요구사항 | 비즈니스 규칙 |
|----|--------|---------|--------------|--------------|
| UC-01 | 회원가입 | High | 이메일·비밀번호·이름 입력 후 계정 생성. 이메일 중복 시 오류 반환 | BR-01, BR-02 |
| UC-02 | 로그인 | High | 이메일·비밀번호 검증 후 JWT Access Token 발급 | BR-03 |
| UC-05 | 할일 등록 | High | 제목(필수), 설명(선택), 종료예정일(선택), 카테고리 선택 후 저장 | BR-07, BR-08 |
| UC-06 | 할일 목록 조회 | High | 카테고리·완료 여부·기간 초과 여부 기준 필터링. 본인 할일만 조회 | BR-09 |
| UC-07 | 할일 수정 | High | 제목·설명·종료예정일·카테고리 변경 가능. 본인 할일만 수정 가능 | BR-09 |
| UC-08 | 할일 완료 처리 | High | 완료 상태 토글(완료 ↔ 미완료). 완료 취소 가능 | BR-10 |
| UC-09 | 할일 삭제 | High | 선택한 할일 영구 삭제. 본인 할일만 삭제 가능 | BR-09 |

**카테고리 관련 MVP 범위 보충**
- 기본 카테고리(Default Category)는 시스템 초기 데이터로 DB에 삽입하여 모든 사용자가 즉시 사용한다.
- MVP에서는 기본 카테고리만 제공하며 사용자 정의 카테고리 생성(UC-04)은 Post-MVP로 이관한다.
- 기본 카테고리는 사용자가 삭제할 수 없다 (BR-04).

### 3.2 Post-MVP 기능 (2차 이후)

| UC | 기능명 | 이유 |
|----|--------|------|
| UC-03 | 개인정보 수정 | 이름·비밀번호 변경은 앱 핵심 동작에 필수가 아님 |
| UC-04 | 카테고리 추가 | 기본 카테고리만으로 최소 동작 가능; 사용자 정의 카테고리는 편의 기능 |
| — | 카테고리 수정·삭제 | UC-04와 함께 2차 릴리즈 |
| — | OAuth Social 로그인 (Google, Facebook) | 인증 확장 — 1차 JWT 안정화 후 적용 |
| — | 다크 모드 | UI 확장 범위 |
| — | 다국어 지원 (i18n) | 서비스 글로벌 확장 시 적용 |
| — | 회원 탈퇴 | 데이터 즉시 삭제 정책과 함께 구현; 2차 릴리즈 |

## 4. 비기능 요구사항

### 4.1 성능

| 항목 | 목표 |
|------|------|
| API 응답 시간 | 일반 요청 95th percentile 500ms 이내 |
| 동시 접속 | 300명 동시 접속 안정 처리 |
| 페이지 초기 로딩 | LCP 3초 이내 (모바일 기준) |

### 4.2 보안

| 항목 | 요구사항 |
|------|---------|
| 인증 | JWT Access Token 기반 인증. 모든 인증 필요 API는 Authorization 헤더 검증 필수 |
| 비밀번호 저장 | bcrypt 등 단방향 해시 알고리즘으로 암호화 저장 (BR-02) |
| 데이터 접근 제어 | 사용자는 자신의 할일·사용자 정의 카테고리만 접근 가능 (BR-03, BR-05, BR-09) |
| 통신 | HTTPS 적용 (프로덕션 환경) |
| SQL Injection | pg 라이브러리의 파라미터화 쿼리 사용 필수 |

### 4.3 확장성

| 항목 | 요구사항 |
|------|---------|
| OAuth 확장 | 2차에서 Google·Facebook OAuth 2.0 소셜 로그인 추가 예정. User 테이블에 `oauth_provider`, `oauth_id` 컬럼 확장을 고려한 설계 필요 |
| 다국어 | 1차 한국어 단일 지원. 2차에서 i18n 라이브러리 도입 예정이므로 UI 문자열 하드코딩 최소화 |
| 다크 모드 | 2차 대응 예정. CSS 변수(Custom Properties) 기반 테마 구조 사전 고려 |

### 4.4 데이터 정책

| 항목 | 요구사항 |
|------|---------|
| 탈퇴 시 데이터 삭제 | 회원 탈퇴 요청 즉시 해당 사용자의 User·Todo·사용자 정의 Category 레코드 모두 삭제 (2차 기능, DB CASCADE 설계는 1차부터 적용) |
| 개인정보 최소 수집 | 이메일·이름·비밀번호만 수집. 불필요한 개인정보 미수집 |

## 5. 기술 스택

### 5.1 확정 기술 스택

| 계층 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 프레임워크 | React 19 + TypeScript | 프론트엔드만 TypeScript 사용 |
| 상태 관리 | Zustand | 클라이언트 전역 상태 |
| 서버 상태 관리 | TanStack Query (React Query v5) | API 캐싱·동기화 |
| 백엔드 런타임 | Node.js + Express (JavaScript) | TypeScript 미사용, CommonJS |
| DB 클라이언트 | pg (node-postgres) | PostgreSQL 연동 필수 라이브러리 |
| 데이터베이스 | PostgreSQL 17 | |
| 인증 | JWT (jsonwebtoken) | 1차 |
| HTTP 로깅 | morgan | Express 요청/응답 로깅 |
| 애플리케이션 로깅 | winston | 구조화 로그 출력 |
| API 문서 | swagger-ui-express | OpenAPI 3.0 스펙 `/api-docs` 에서 제공 |
| 백엔드 테스트 | Jest + supertest | 단위·통합 테스트, 커버리지 80% 이상 |

### 5.2 인증 아키텍처

```
1차 (MVP)
  클라이언트 → POST /auth/login → 서버 검증 → JWT Access Token 발급
  클라이언트는 이후 모든 요청 헤더에 Authorization: Bearer <token> 포함
  JWT Access Token은 Zustand 스토어(메모리)에 저장 — 페이지 새로고침 시 재로그인 필요

2차 (Post-MVP)
  OAuth 2.0 소셜 로그인 추가 (Google, Facebook 우선)
  기존 JWT 발급 플로우 유지 + OAuth 콜백 후 동일 JWT 발급 방식으로 통합
  User 테이블에 oauth_provider, oauth_id 컬럼 추가
```

### 5.3 pg 라이브러리 사용 원칙
- 모든 PostgreSQL DB 연동은 반드시 `pg` 라이브러리를 통해 수행한다.
- SQL 쿼리는 파라미터화 쿼리(`$1`, `$2` ... 플레이스홀더)를 사용하여 SQL Injection을 방지한다.
- ORM(Sequelize, Prisma 등) 사용 금지.

## 7. 데이터 모델 요약

> 상세 DDL은 별도 데이터베이스 설계 문서에서 관리한다.

### users 테이블

| 컬럼명 | 타입 | 제약 | 설명 |
|--------|------|------|------|
| id | SERIAL | PK | 사용자 고유 식별자 |
| email | VARCHAR(255) | UNIQUE, NOT NULL | 로그인 ID |
| password | VARCHAR(255) | NOT NULL | bcrypt 해시 저장 |
| name | VARCHAR(100) | NOT NULL | 표시 이름 |
| created_at | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 가입일시 |

> 2차 OAuth 확장 시 `oauth_provider VARCHAR(50)`, `oauth_id VARCHAR(255)` 컬럼 추가 예정

### categories 테이블

| 컬럼명 | 타입 | 제약 | 설명 |
|--------|------|------|------|
| id | SERIAL | PK | 카테고리 고유 식별자 |
| name | VARCHAR(100) | NOT NULL | 카테고리명 |
| is_default | BOOLEAN | NOT NULL, DEFAULT FALSE | 기본 카테고리 여부 |
| user_id | INTEGER | FK → users.id, NULL 허용 | NULL이면 기본 카테고리 |

- 동일 user_id 범위 내에서 name 중복 불가 (BR-06). `user_id`가 NULL인 기본 카테고리끼리도 name 중복 불가.
- 삭제 정책: `user_id`에 `ON DELETE CASCADE` 적용 → 사용자 삭제 시 사용자 정의 카테고리 자동 삭제

### todos 테이블

| 컬럼명 | 타입 | 제약 | 설명 |
|--------|------|------|------|
| id | SERIAL | PK | 할일 고유 식별자 |
| user_id | INTEGER | FK → users.id, NOT NULL | 소유 사용자 |
| category_id | INTEGER | FK → categories.id, NOT NULL | 소속 카테고리 |
| title | VARCHAR(255) | NOT NULL | 할일 제목 (BR-08) |
| description | TEXT | NULL 허용 | 상세 설명 |
| due_date | DATE | NULL 허용 | 종료예정일 |
| is_completed | BOOLEAN | NOT NULL, DEFAULT FALSE | 완료 여부 |
| created_at | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 등록일시 |
| updated_at | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 최종 수정일시 |

- `user_id`에 `ON DELETE CASCADE` 적용 → 사용자 삭제 시 할일 자동 삭제
- 기간 초과(Overdue) 판별: `due_date < CURRENT_DATE AND is_completed = FALSE` (애플리케이션 또는 쿼리 레벨)

## 8. REST API 엔드포인트 목록 (개요)

> 상세 요청/응답 스펙(파라미터, 바디, 상태코드, 에러 형식)은 별도 API 명세 문서에서 관리한다.

### 인증 (Authentication)

| 메서드 | 엔드포인트 | 설명 | 인증 필요 |
|--------|-----------|------|----------|
| POST | /api/auth/register | 회원가입 | 없음 |
| POST | /api/auth/login | 로그인 및 JWT 발급 | 없음 |

### 사용자 프로필 (User Profile) — Post-MVP

| 메서드 | 엔드포인트 | 설명 | 인증 필요 |
|--------|-----------|------|----------|
| GET | /api/users/me | 내 프로필 조회 | JWT |
| PATCH | /api/users/me | 내 프로필 수정 | JWT |
| DELETE | /api/users/me | 회원 탈퇴 | JWT |

### 카테고리 (Category)

| 메서드 | 엔드포인트 | 설명 | 인증 필요 |
|--------|-----------|------|----------|
| GET | /api/categories | 카테고리 목록 조회 (기본 + 내 카테고리) | JWT |
| POST | /api/categories | 사용자 정의 카테고리 생성 (Post-MVP) | JWT |
| PATCH | /api/categories/:id | 사용자 정의 카테고리 수정 (Post-MVP) | JWT |
| DELETE | /api/categories/:id | 사용자 정의 카테고리 삭제 (Post-MVP). 기본 카테고리 삭제 요청 시 403 Forbidden 반환 (BR-04) | JWT |

### 할일 (Todo)

| 메서드 | 엔드포인트 | 설명 | 인증 필요 |
|--------|-----------|------|----------|
| GET | /api/todos | 내 할일 목록 조회 (필터: category_id, is_completed, overdue) | JWT |
